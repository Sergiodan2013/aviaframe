## [D-001] Архитектура: монорепозиторий vs отдельные репозитории

**Дата:** 2026-01-26
**Статус:** ✅ Принято

**Проблема:**
Проект состоит из трёх частей (backend, portal, widget). Как их хранить и деплоить?

**Варианты:**
- A) Три отдельных репозитория — независимые деплои, но сложная синхронизация версий
- B) Монорепозиторий с npm workspaces — единый источник правды, общие типы и конфиги

**Выбрано:** B — монорепозиторий с npm workspaces

**Почему:**
Widget, portal и backend тесно связаны (API-контракт, типы, конфиги). Монорепо упрощает рефакторинг, CI и управление зависимостями на ранних стадиях MVP.

## [D-002] DRCT-интеграция: прямая vs через адаптер

**Дата:** 2026-01-26
**Статус:** ✅ Принято

**Проблема:**
Как интегрироваться с DRCT API? Можно дать виджету прямой доступ или проксировать через backend.

**Варианты:**
- A) Виджет напрямую вызывает DRCT — проще, но credential leak и нет контроля
- B) Всё через backend-адаптер — DRCT-токены только на сервере

**Выбрано:** B — DRCT Adapter на backend

**Почему:**
DRCT-токены не должны попадать в браузер. Адаптер даёт: rate limiting, idempotency, tenant isolation, audit logging. Это hard security requirement.

## [D-003] Multi-tenancy: одна БД vs отдельные БД на тенанта

**Дата:** 2026-01-26
**Статус:** ✅ Принято

**Проблема:**
Как изолировать данные между агентствами (тенантами)?

**Варианты:**
- A) Отдельная БД на тенанта — максимальная изоляция, но сложный ops и дорого
- B) Одна БД с tenant_id колонкой + RLS (Row Level Security) в Supabase

**Выбрано:** B — single DB + RLS

**Почему:**
Supabase RLS даёт надёжную изоляцию без ops-оверхеда нескольких БД. Достаточно для MVP и большинства production-сценариев. Переход на схему-per-tenant возможен при необходимости.

## [D-004] Workflow-движок: custom vs n8n

**Дата:** 2026-01-26
**Статус:** ✅ Принято

**Проблема:**
Как управлять сложными workflow (booking lifecycle, уведомления, retry-логика)?

**Варианты:**
- A) Custom code в backend — полный контроль, но сложная поддержка и нет визуализации
- B) n8n как workflow-оркестратор — визуальный, self-hosted, интегрируется с webhook

**Выбрано:** B — n8n

**Почему:**
n8n снижает сложность поддержки сложных потоков (issue → notify → audit). Легко итерировать workflow без деплоя backend. Self-hosted, данные остаются внутри инфраструктуры.

## [D-005] Auth: custom vs Supabase Auth

**Дата:** 2026-01-26
**Статус:** ✅ Принято

**Проблема:**
Как управлять аутентификацией пользователей (magic link, JWT, RBAC)?

**Варианты:**
- A) Custom JWT / session система — максимальный контроль, высокая стоимость разработки
- B) Supabase Auth с magic link + кастомные роли в БД

**Выбрано:** B — Supabase Auth

**Почему:**
Supabase Auth даёт magic link, JWT, управление сессиями из коробки. Кастомные роли (super_admin, agency_admin, agency_manager) реализуются через metadata и RLS-политики. Ускоряет MVP.

## [D-007] Widget: vanilla JS vs framework

**Дата:** 2026-01-26
**Статус:** ✅ Принято

**Проблема:**
На чём строить embeddable widget?

**Варианты:**
- A) React / Vue компонент — богатая экосистема, но тяжёлый bundle и конфликты с host-сайтом
- B) Vanilla JS — легковесный, нет зависимостей, нет конфликтов версий

**Выбрано:** B — Vanilla JS

**Почему:**
Widget встраивается на сторонние сайты через `<script>` тег. Vanilla JS гарантирует отсутствие конфликтов с чужими фреймворками и минимальный размер bundle. Это критично для embeddable-продукта.

## [D-008] Idempotency: для каких операций обязательна

**Дата:** 2026-01-26
**Статус:** ✅ Принято

**Проблема:**
Какие DRCT-операции требуют idempotency-защиты?

**Решение:**
Обязательна для всех state-changing операций: `order_create` и `issue`. Search и price — идемпотентны по природе (только чтение). Cancel — отдельная логика.

**Почему:**
Двойной booking или двойной issue — критическая ошибка для авиабилетов. Idempotency-key + запись в БД до вызова DRCT — единственная надёжная защита от retry-дублирования.

## [D-009] Rate limiting DRCT: in-process vs Redis

**Дата:** 2026-03-29
**Статус:** ✅ Принято

**Проблема:**
Нужен rate limiter для DRCT (1 rps). Как реализовать?

**Варианты:**
- A) Redis-based limiter (ioredis + sliding window) — точный, работает на нескольких инстансах
- B) In-process с `bottleneck` — простой, без доп. инфраструктуры

**Выбрано:** B — `bottleneck` (in-process) для MVP

**Почему:**
На текущем этапе backend запущен в одном инстансе на Railway. In-process limiter достаточен. При переходе к горизонтальному масштабированию — перейти на Redis. Зафиксировать как точку пересмотра.

## [D-010] Circuit breaker: opossum

**Дата:** 2026-03-29
**Статус:** ✅ Принято

**Проблема:**
Какой circuit breaker использовать для DRCT?

**Варианты:**
- A) Кастомный (счётчик ошибок + флаг) — просто, но нет half-open логики
- B) `opossum` — стандартная библиотека, активно поддерживается, half-open из коробки

**Выбрано:** B — `opossum`

**Почему:**
Half-open состояние критично для правильного восстановления после сбоя DRCT. `opossum` даёт это из коробки с event hooks для логирования.

## [D-011] Логирование: pino вместо console.log

**Дата:** 2026-03-29
**Статус:** ✅ Принято

**Проблема:**
Текущие `console.log` неструктурированны — нет контекста (tenant_id, order_id), нет уровней, нельзя парсить в production.

**Выбрано:** `pino` — самый быстрый structured logger для Node.js, JSON-output в production, pretty-print в dev.

**Почему:**
`pino` в 5-10x быстрее winston/bunyan. JSON-логи совместимы с любым log aggregator (Datadog, Logtail, Railway logs).
